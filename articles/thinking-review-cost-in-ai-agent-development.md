## はじめに

AI Agentの開発によって爆発的にコードのアウトプットの量が増えました。  
人がゼロから書くよりも圧倒的速くたたき台が出てくるので、実装の初速は確実に上がりました。  
その意味では、継続的な開発サイクル全体のスピードも上がっている実感があります。  
ただ一方で、「コードが生成されること」と「本番環境にデプロイできること」のあいだにはまだ距離があると感じており、その中で、時間のかかるポイントの1つがレビューだと思っています。

ここで「レビューがボトルネックだ」と言い切ってしまうと、少し語弊がある気もしています。  
レビュー自体が悪いわけではなく、本番投入までのフローを成立させるうえで、今はレビューに多くの仕事が集まっているように見える、くらいの感覚です。

だからこそ、私自身は単純にレビューを減らすとかやめるという判断にはまだ踏み切れていません。  
自分の中に「レビューには重要な役割があるはずだ」という感覚が残っています。

この記事では、まずコードレビューが何を担っているのかを整理してみます。  
その上で、その役割のうち何がレビュー以外に逃がせるのか、どこを前倒ししたり自動化したりできるのかを考えてみたいです。

## コードレビューの役割はなにか

コードレビューは、単にコードの正しさを確認する場ではないはずです。  
実際には、いくつか別の役割がまとめて流れ込んでいます。

- バグや事故の検知
- 仕様理解の補完
- 設計判断の整合
- 品質基準の維持
- 将来の保守のための説明責任

たとえば「この条件分岐で漏れはないか」を見るのは、バグや事故の検知です。  
「そもそもこの変更はどんな背景で必要になったのか」を追うのは、仕様理解の補完です。  
「命名や責務の置き方はこのコードベースに合っているか」は、設計判断の整合に近いです。

さらに、レビューはチームの品質基準を揃える役割も持っています。  
テストを書くのが当然なのか、監視まで含めて変更と見なすのか、ドキュメント更新をどこまで求めるのか。  
そうした暗黙の期待が、レビューコメントを通して共有されていることも多いです。

そして、あとから見返したときに「なぜこうなっているのか」をたどれる状態を作ることも、レビューの役割の1つだと思います。  
つまりレビューは、品質確認、知識共有、設計の合意、変更理由の記録をまとめて引き受けています。

## コードレビューになぜ時間がかかるのか

AI Agent によって実装の初速が上がるほど、レビュー側に求められる処理量とのギャップが見えやすくなります。  
自分が感じている要因は、いまのところ次のようなものです。

- 変更量が増えた
- 実装速度に対して理解速度が追いつかない
- 自動で確認できることまで人が読んでいる
- 設計相談と実装確認が同じ PR に乗っている

まず単純に、AI Agent が出す差分は大きくなりがちです。  
差分が大きいほど、レビュアーは「何が変わったか」より前に「どこから読めばいいか」を判断する必要が出てきます。  
それだけで認知負荷が高くなります。

次に、実装は速くなっても、内容を理解する速度まで同じように上がるわけではありません。  
コードを出す速度が上がっても、その変更が既存仕様や周辺機能にどう効くのかを読み解く速度は、そこまで急には上がりません。  
結果として、アウトプット量の増加がそのままレビュー待ちの増加に見えやすくなります。

また、本来は lint、型、テスト、CI、静的解析に任せられる確認まで人がやっている場合、レビューの仕事はさらに膨らみます。  
加えて、設計の論点が事前に整理されないまま PR へ流れ込むと、実装レビューの場が設計会議の代わりにもなってしまいます。

## コードレビューの役割は何に分割できるか

レビューが重いならレビューを減らす、ではなく、レビューに集まっている役割を分解して移せないかを考えたいです。  
全部を人が抱える必要はないはずです。

- テスト、型、lint、CI、静的解析に寄せる
- 設計メモや Issue で事前に相談する
- PR 本文やテンプレートで背景共有を先に済ませる
- スクリーンショット、動作確認手順、差分分割で認知負荷を下げる

たとえば、機械的に検出できるものは機械に寄せるのが自然です。  
フォーマット、命名の一貫性、型の不整合、明らかなテスト不足などが毎回レビューに出てくるなら、それはレビューの前に落とせる余地があります。

仕様や設計の相談も、できれば実装 PR とは分けたいです。  
変更の方向性に合意がない状態で大量の差分だけが届くと、レビュアーはコードを読みながら設計判断まで引き受けることになります。  
事前メモ、Issue、軽い ADR のような形で論点が整理されていれば、それだけでレビューの読み方はかなり変わります。

背景共有も同じです。  
何を変えたかだけでなく、なぜ変えたのか、どこを見てほしいのか、どこまで確認済みなのかが PR に書かれていると、レビュアーは確認すべき点に集中しやすくなります。

## チームの判断基準を言語化し、AIが出すPRの認知負荷を下げる

レビューコストはレビュアーだけの問題ではなく、差分を出す側の工夫でもかなり変わるはずです。  
ただ、ここを個人のうまさだけに任せても再現性は出ません。  
必要なのは、レビューで毎回やりとりされている判断基準を、チームの言葉として前に出していくことだと思っています。

たとえば、次のようなものです。

- このチームがよしとする責務分割
- 命名や設計で避けたいこと
- このドメインで壊しやすいポイント
- 変更時に最低限そろっていてほしい情報
- 仕様に書き切れていない背景や前提

こうした判断基準が暗黙知のままだと、AI Agent が出した差分を見るたびに、レビュアーはコードだけでなく前提まで読み解く必要があります。  
差分が大きいほど、その負荷は急激に上がります。

逆に、判断基準が言語化されていれば、PR はもっと読みやすくできます。  
PR を小さく保つ、変更理由と見てほしい観点を書く、セルフレビューで潰した論点を先に残す、といった工夫も、個人技ではなくチームの型として扱えるようになります。

特に AI Agent が生成したコードは、一見もっともらしくても文脈に合っていないことがあります。  
だからこそ、生成されたコードそのものより、そのコードを採用した意図や前提、どの判断基準に沿っているのかを補足できる状態が重要になるはずです。

## 言語化の先で、レビューはどこまで必要なのか

ここまで考えてくると、そもそもレビューをどこまで残したいのか、という問いが出てきます。  
正直に言うと、自分は「それでもレビューを残したい」と強く思っているわけではありません。

むしろ、チームのコンテキストや判断基準の言語化が進んだ先では、レビューという形式そのものをかなり薄くできるのではないか、と感じています。  
lint、テスト、型、静的解析、設計共有、PR の型が整っていけば、今レビューで引き受けている仕事の多くは別の場所へ逃がせるかもしれません。

もちろん、すべてが自動化できるとはまだ言えません。  
ただ、いまレビューに見えている仕事のかなりの部分は、レビューでなければならない理由が弱いのではないか、とも思っています。

## レビューの代わりに必要な共有は何か

一方で、レビューが薄くなったとしても、共有そのものが不要になるわけではないはずです。  
現状のコードや仕様だけでは拾いきれない背景や、コード化されていないドメイン理解はまだ多く残っています。

その意味では、残したいのはレビューという形式ではなく、チームのコンテキストを揃える時間なのかもしれません。

- コードリーディングの時間を取る
- 仕様や設計の読み合わせを行う
- 変更理由や前提をドキュメントとして残す
- ドメイン理解に必要な背景を継続的に共有する

いまレビューが担っている役割の中には、品質確認だけでなく、背景共有や学習の場としての側面もあります。  
もしレビューを薄くしていくなら、その役割をどこで代替するのかは別に考える必要があります。

## この仮説を確かめたい

ここまでの話を書きながら、これは既存の開発フローの延長にすぎないのかもしれない、とも感じています。  
つまり、いまある「人が実装して、人がレビューして、本番へ出す」という流れを前提に、その中でレビュー負荷を下げようとしているだけなのではないか、という感覚です。

まだ自分は、AI Agent に合わせた開発フローをゼロから設計できているわけではありません。  
レビューのあり方そのものを組み替える余地も、本当はもっとあるのかもしれません。

それでも、lint、テスト、ドキュメント、設計共有を整えた先で、AI Agent がどんなコードを出し、レビューコストがどれくらい下がり、開発サイクルがどんな速度になるのかは見てみたいと思っています。  
そして、その先で本当にレビューがどこまで不要になるのか、代わりにどんな共有の場が必要になるのかを確かめたいです。

## おわりに

いま自分が考えているのは、レビューを守りたいという話ではありません。  
レビューでやっていることを分解し、必要なものは別の形へ移し、それでも残る共有だけを見極めたいという話です。

この方向が本当に正しかったのかは、整備を進めた先で初めて見えてくるのだと思います。  
それでも、少なくとも間違った方向ではないように感じています。  
なので、まずは一度この仮説を信じて進んでみたいです。
