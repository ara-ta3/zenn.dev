# Derive Article Direction

## Overview

- 既存記事から「よく書いているテーマ」「既に書いた導入」「まだ掘れていない周辺論点」を抽出する。
- 新しい記事の方向性は、必ず既存記事の根拠とセットで提案する。一般論だけで埋めない。
- `articles/` と `hashnodes/` を優先して読む。ユーザーが別のパスを指定したらそちらを優先する。

## Workflow

### 1. 対象記事を絞る

- ユーザーが指定したテーマ、技術、既存記事を起点に対象範囲を決める。
- `rg --files articles hashnodes | rg '<keyword>'` のように候補を絞る。
- 候補が多すぎる場合は topic、タイトル、導入文が近いものから 3-8 本までに縮める。
- ユーザーが明示したファイルは必ず残す。

### 2. シグナルを素早く集める

- まず `python3 skills/derive-article-direction/scripts/collect_article_signals.py --query <keyword> <paths...>` を実行する。
- 候補を広く眺めたいときは `--limit` で件数を絞る。
- 未公開記事や下書きも見たいときだけ `--include-unpublished` を使う。
- フルテキストを読む前に、タイトル、topics、導入段落、H2/H3、まとめを優先して確認する。

### 3. 差分と空白を見つける

- 既存記事の共通点を整理する。
- 読者レベル、導入の型、扱っている環境、見出しの流れ、締め方を比較する。
- 既に書いていることと、次の一歩として自然なことを切り分ける。
- 次の切り口を優先する。
  - 入門記事の次に来る実践編
  - セットアップ記事の次に来るトラブルシュート
  - 個別ツール記事の次に来る比較記事
  - 手作業の記事の次に来る自動化記事
- タイトル、読者への約束、見出し構成の3つが同時に重複する提案は避ける。

### 4. 方向性メモを書く

- 出力は `references/output-format.md` の形を基本にする。
- 各提案には 1-3 本の根拠記事を添える。
- 「何を書くか」だけではなく「なぜ今それを書くと既存記事群とつながるか」を明記する。
- 曖昧さが残る場合は、仮説と確認ポイントを分けて書く。

## Heuristics

- 少なくとも次のどれか1つは変える。
  - 読者レベル
  - 問題設定
  - 対象環境
  - 比較軸
  - 運用フェーズ
- 強い方向性は「既存記事が成功したあとに次に困ること」から作る。
- このリポジトリでは次の型が多いので、関連する次の一手を優先する。
  - getting started / セットアップ
  - コマンドやオプションの深掘り
  - 実際のエラー起点のトラブルシュート
  - 開発環境や運用改善
- 関連記事が多い場合は、次のどれかに寄せる。
  - 続編
  - 比較
  - 落とし穴
  - 自動化
  - 別言語 / 別フレームワークへの横展開
- 関連記事が少ない場合は、総論よりも「前回の続き」を優先する。

## Output Guardrails

- フル記事を書き始めない。まず方向性に絞る。
- 既存記事で既に説明済みの内容を、新規性があるように見せない。
- 既存に入門記事があるなら、差分のない「○○入門」は提案しない。
- ライターがそのまま構成に起こせる粒度まで具体化する。

## Resources

- `scripts/collect_article_signals.py`: Markdown 記事から frontmatter、見出し、導入文、簡易スコアを抽出する。
- `references/output-format.md`: 記事の方向性メモを書くときの標準フォーマット。
